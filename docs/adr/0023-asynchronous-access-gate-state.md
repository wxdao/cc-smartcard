# Asynchronous Access Gate State

Access Gate movement is asynchronous: `open()` and `close()` accept a command immediately, while `getState()` and `access_gate_state_changed` report progress through `closed`, `opening`, `open`, `closing`, `obstructed`, and `unpaired` states. A command opposite to the current movement reverses it from its current progress, while a repeated command in the same direction succeeds idempotently; commands are not queued. If a player, creature, armour stand, or vehicle obstructs closing barrier leaves, the gate reports the obstruction and automatically reverses until fully open, leaving retry or alarm policy to the controlling program. `obstructed` is latched for two server ticks while reversal proceeds so synchronous state queries have a bounded opportunity to observe it; the dedicated obstruction event remains the durable reactive signal. Dropped items, experience orbs, and projectiles do not trigger this reversal because they would make a trivial gate bypass. Blocking until an animation completes was rejected because a gate is a shared actuator observed and controlled by potentially multiple computers, and movement may be delayed indefinitely by an obstruction.
